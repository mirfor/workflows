Weaver — backend silnik Agentów: kierunek i status

Krótkie podsumowanie tego, gdzie jesteśmy z backendowym silnikiem dla Weaver Agentów. Dla zespołu — żebyśmy zgadzali się co budujemy, dlaczego tak, co już działa i co dalej. Zgodne z Weaver vocabulary.

Po co to robimy

Weaver to Business Agents OS — klient kupuje i operuje Agentami, którzy wykonują biznesowe zadania end-to-end (zatwierdzanie wydatków, kwartalne rozliczenia, onboarding, etc.). Agent łączy AI reasoning, użycie Tools i ludzkie decyzje; uruchamia się niezawodnie pod naszą orkiestracją. Świadomie odrzucamy słownik BPM ("process", "workflow", "case") — patrz vocabulary.

Backendowy silnik (to repo) bierze Agent Blueprint (graf zaprojektowany w Agent Studio) i sprawia, że można go uruchamiać jako Engagement w Temporalu — niezawodnie, deterministycznie, audytowalnie.

Dlaczego nie własny DSL + interpreter

W fazie projektowej rozważaliśmy podejście DSL + interpreter: własny język opisu Agenta w JSON-ie + uniwersalny worker Pythona, który by go parsował i wykonywał krok po kroku. Odrzucone z tych powodów:

  • Interpreter dubluje Temporal SDK — Temporal już daje deterministyczny model wykonania z replay/versioning; własna warstwa interpretacji ponad to komplikuje debug i tracking
  • DSL = ciągły rozwój własnego języka — każda nowa konstrukcja wymaga rozszerzenia parsera + walidatora + interpretera + dokumentacji "co znaczy każdy tag"
  • Zerowa type safety — JSON jest stringly-typed; błędy wyłapuje się dopiero w runtime
  • Replay / debug nie działa po stronie domeny — Python stack trace pokazuje interpreter, nie kod biznesowego Agenta
  • Standard branżowy istnieje — patrz słowniczek niżej (CNCF Serverless Workflow 1.0)

Wybraliśmy: codegen Pythona dla Temporala. Każdy Agent Blueprint × wersja produkuje fizyczny plik .py — taki, jakby ktoś go napisał ręcznie z dekoratorami Temporala.

Uwaga o "migracji". W Weaverze trwa równoległa migracja terminologii w trzech frontendowych repos: BPM-owe nazwy ("Process", "Case", "Gateway") są wymieniane na słownik Agent ("Agent", "Engagement", "Decision Point") — patrz weaver-root/docs/content/architecture/vocabulary/migration.md. Nie dotyczy to tego repo — backend jest greenfield, nie ma istniejących Agentów do przenoszenia.

Używamy go jako kanonicznej reprezentacji Agent Blueprintu między Designerem a generatorem Pythona — żeby nie wymyślać własnego formatu.

Jak rysunek z Designera staje się Pythonem

Konkretna mechanika pipeline'u — najważniejsza techniczna treść tego dokumentu.

1Agent Studio (Designer) → React Flow JSON

Użytkownik rysuje Agent Blueprint w przeglądarce: Core Skills, Tools, Specialized Agents jako node'y na canvas; połączenia między nimi; konfiguracja per node (np. który Tool wywołać, jakie warunki, jakie inputy/outputy). Po kliknięciu Save Draft design serializuje się do React Flow JSON: { nodes: [...], edges: [...] } z metadanymi pozycji i danych formularzy.

2Mapper: React Flow JSON → CNCF Serverless Workflow IR JSON

Moduł mapper/reactflow_to_cncfsw.py dokonuje deterministycznej, jednokierunkowej konwersji. Każdy typ node'a w React Flow mapuje się na konkretny task type ze standardu CNCF SW 1.0. Wykrywa trigger (node bez incoming edges), oblicza branch ownership (które node'y należą do której gałęzi switch / fork / try → catch), buduje topologiczną sekwencję. Wynik to ustrukturyzowany IR JSON — kanoniczna semantyka Agent Blueprintu.

3Walidator IR — 6 kategorii reguł

Moduł validator/validator.py sprawdza poprawność IR przed generacją: (A) struktura grafu (duplikaty IDs, container body non-empty), (B) handles/edges (switch.default, declared handles), (C) registry funkcji (czy wywoływane Tools/Specialized Agents istnieją w manifeście), (D) schemy I/O, (E) polityki Temporala (retry mapping, czy nie używa pól bez Temporal mapping jak jitter, when, limit.duration), (F) zgodność ze spec CNCF SW. Severity: error blokuje publish; warning tylko notyfikuje.

4Generator: IR → Python AST → .py

Moduł generator/codegen.py bierze IR i buduje plik .py przez Python ast module (nie string templating — branżowy standard, type-safe, formatter-friendly). Mapping per task type:

  • call → await workflow.execute_activity(...) z timeout + retry policy
  • switch → if/elif/else z body inline (sprawdzone: tylko właściwa gałąź się wykonuje)
  • for → for <each> in _eval(<in>, ctx):
  • fork → asyncio.gather(*tasks) lub asyncio.wait(FIRST_COMPLETED)
  • try/catch → try: ... except ApplicationError as e: ...
  • wait → await workflow.sleep(timedelta(...))
  • itd. dla wszystkich 12 task types

Warunki w switch i if ewaluują się przez wbudowany helper _eval() z compiled JQ programs cache (JQ jako standard expression language). Po zbudowaniu AST: ast.unparse() → string Pythona → black.format_str() dla finalnego stylu.

5Plik .py + manifest per Tenant

Wynik to generated/<tenant>/workflows/<snake_id>__v<n>.py z headerem:

# Generated from Blueprint <tenant>/<id> v<n> at <timestamp>
# Source hash: <sha256-of-IR>
# DO NOT EDIT — regeneruj przez generator

Source hash to SHA256 z deterministycznie znormalizowanego IR — gwarantuje idempotencję regeneracji (ten sam IR → identyczny .py). Plik dodawany do generated/<tenant>/manifest.json z active_version, deprecated_versions, build_id_lineage.

6Worker per Tenant importuje wygenerowane pliki

Worker uruchamiany z uv run python worker.py --tenant <id>. Czyta generated/<tenant>/manifest.json, importuje tylko active wersje workflowów tego Tenanta, rejestruje activities, łączy się z Temporalem (namespace = tenant_id, task queue = weaver-<tenant>). Klient startuje Engagement przez start_workflow — Temporal kieruje go do właściwego workera.

Co dzieje się po zmianie Agent Blueprintu (deploy)

Nowa wersja Blueprintu = nowy plik .py + git commit + CI build z nowym Worker Versioning Build ID + rolling deploy. Stare Engagementy kończą się na poprzednim Build ID, nowe trafiają na nowy. Zero przerwy. Pełen audyt — każda wersja w git history forever.

Co już działa

Cały proces działa od początku do końca — sprawdzone na żywym Temporal Server z dwoma Tenantami. Nie tylko unit testy: realne Engagementy startują, wykonują się, kończą z poprawnym wynikiem.

1. Pełna ścieżka E2E

Agent Blueprint w blueprints/<tenant>/<id>/v<n>/reactflow.json → mapper → IR JSON (cncf-sw.json) → walidator → generator → .py → manifest → Worker → Temporal → wykonanie. Każda forma Blueprintu przechowywana w git po publikacji (3 formy obowiązkowe — patrz ADR-002).

2. Każdy Tenant ma odizolowaną przestrzeń

  • Pliki: blueprints/<tenant>/<agent_id>/v<n>/ i generated/<tenant>/...
  • Każdy Tenant: własny Temporal namespace, własna kolejka weaver-<tenant>, własny Worker
  • Manifesty Tenantów disjoint w blueprint_ids — Worker jednego Tenanta nie widzi Agentów innego
  • E2E zweryfikowane (test F5.6): start hello w namespace demo nie znajduje workera (Agent hello istnieje tylko u acme)

4. Compliance gate — nie pozwala "olewać" decyzji

30 decyzji projektowych z fazy designerskiej (docs/SESSION_STATE.md) ma swoje testy w tests/test_compliance.py. Mapowanie w docs/COMPLIANCE.md. CI blokuje merge, gdy compliance test fail. PR review pierwsze pytanie: "który compliance test się zmienił?". Aktualnie 34/34 testy passing.

O czym warto pogadać na meetingu

Decyzje produktowe / operacyjne (odłożone w fazie projektowej)

  • Audience Agent Studio — kto realnie projektuje Agenty? Developer / analityk / nietechniczny user. Wpływa na ergonomię (raw JQ vs structural builder), ile Core Skills jest dostępnych w palecie.
  • Target deployment — Cloud Run, GKE, self-hosted? Temporal Cloud czy własny? Wpływa na Worker partitioning, secrets, observability.
  • Skala docelowa — ile Agent Blueprintów per Tenant (setki vs tysiące) zmienia strategię ładowania workerów.
  • Implementacja Tools — kto pisze poszczególne integracje? Mamy szablon activities/tools/<integration>.py z manifestem; potrzebujemy ownerów per integracja.
  • RBAC / multi-role UI — Tenant admin / Client Org operator / Designer / Reviewer. Nie było przedmiotem sesji projektowej, wymaga osobnej.

Architektoniczne — chciałbym potwierdzenie

  • Trzymamy się CNCF Serverless Workflow 1.0 jako kanonicznego IR JSON pomiędzy Designerem a generatorem (zamiast własnego DSL — porzucone).
  • Generujemy fizyczny .py per Agent Blueprint × wersja (nie interpretujemy IR w runtime).
  • Worker Versioning Build ID Temporala dla wersjonowania (nie workflow.patched()).
  • Per-Tenant fizyczna izolacja (osobny Temporal namespace + DB), per Client Org logiczna (row-level + Search Attribute).

Demo flow (5 min)

  1. 30 decyzji w SESSION_STATE.md, 6 ADR, mapowanie na compliance testy
  2. uv run pytest tests/test_compliance.py -v → 34 passing
  3. Bulk regeneracja: uv run python -m scripts.regenerate_all → 5 Blueprintów × 2 Tenanty
  4. Pokażę wygenerowany .py z headerem DO NOT EDIT
  5. temporal server start-dev, dwóch workerów (per Tenant), uruchomię Engagement z Pythona
  6. Cross-tenant isolation: hello nie wykonuje się w demo namespace
